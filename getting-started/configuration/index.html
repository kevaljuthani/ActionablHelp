# Configuration

We will be looking at various configurations that are available to users and developers for customizations and tweaks in Actionabl

 app: {
        name: 'actionabl',
        desc: 'Actionabl',
        version: '1.0'
    }

Inside app object, one can configure name, description and version of the application.

system: {
        user_id: 'actionabl',
        simulation_mode: false,
        local_email_mode: false,
        debug_mode: false,
        unit_test_mode: false
    }
  • simulation_mode: As the name suggests, application will be running in simulation mode. For example, if set to true, the process where there is a email sending task present will be performed but it won't be actually sending an email but that email will be stored in temporary runtime folder in html format
  • debug_mode: This allows to debugs the deployed process. One can go to Process View and skip / put a breakpoint at a task
 security: {
        encrypt_repository: false,
        encrypt_data: false,
        password_expiry_period: '99y',
        reset_password_expiry: '2h',
        pwd_history_check: 2,
        email_link_token_expiry: '30d',
        session_max_idle_time: 1800,
        session_check_frequency: 600,
        use_captcha: false,
        contentPolicy: {
            defaultSrc: ["'self'"],
            fontSrc: ["'self'", "data:", "https://*"],
            scriptSrc: ["'self'", "'unsafe-eval'", "http://*", "'unsafe-inline'", "blob"],
            frameSrc: ["'self'", "data:", "http://*", "https://*"],
            styleSrc: ["'self'", "https:", "'unsafe-inline'"],
            imgSrc: ["'self'", "data:"],
            connectSrc: ["'self'", "http://localhost:5005/"],
            //frameAncestors: ["http://localhost:3001/", "https://*"],
        }
    }
  • encrypt_repository: If set to true, the components data stored in the file system, such as, connectors, record, fields, etc. will be stored in encrypted form
  • encrypt_data: If set to true, the process instance level data will be stored in an encrypted format in the database
  • password_expiry_period: Period after which account's password will be expired
  • reset_password_expiry: When user requests Forgot Password, Actionabl sends link to the user which allows user to reset the password. The time for which this link is valid is determined by this parameter. Default value is 2 hours
  • pwd_history_check: Current password that is being changed is checked against number of previous passwords
  • email_link_token_expiry: This is used to when Actionabl sends a link to customer for some action from customer side. This parameter determines the period for which this link is valid. Default value is 30 days
  • session_max_idle_time: Default value is 1800 seconds (30 minutes). If user does not perform any action on browser for more than this value, user is logged out
  • session_check_frequency: The interval at which system will check whether the session is idle. Default value is 600 seconds
  • use_captcha: If set to true, login screen will present captcha validation fields for additional safety
db: {
        host: 'localhost',
        user: 'admin',
        dba_user: 'postgres',
        database: 'actionabl',
        max: 10,
        connectionTimeoutMillis: 10000,
        idleTimeoutMillis: 30000
    }
  • host: Provide host address for database server
  • user: Provide user who has privileges to access the database
  • dba_user: User who has right to create database
  • database: Provide name of the database
  • connectionTimeoutMillis: Connection timeout in milliseconds
  • idleTimeoutMillis: Idle database timeout in milliseconds
db_setup: {
        run_setup: false,
        recreate: false,
    }
  • run_setup: If set to true, server will create fresh setup of application
  • recreate: If set to true, server will clear all the data from the database and recreate the database with default data
 component_cluster_setup: {
        default: 'invoice_query',
        invoice_query: true
    }
  • default: Provide name of the process you want to make as default. Any connectors or records or rules created will be stored in Components folder of this process
  • invoice_query: It is an example process name. You can have any number of process added here. If set to true, the process is enabled
 jobs: {
        start_standalone_bots: true,
        default_max_task_attempts: 3,
        default_run_interval: 60000,
        default_max_concurrent_runs: 25,
        cronProcessor: { 
            start: true,
            run_interval: 60000,
        }
    }
  • start_standalone_bots: If set to true, the Actionabl will start the standalone jobs also when the server is started
  • default_max_task_attempts: Provide default maximum task attempts in case of task failure/error
  • default_run_interval: Default value is 1 minute. It sets the interval between two job runs (interval after which same job/task is run again). This can be overridden at job / task level
  • default_max_concurrent_runs: Sets the maximum limit on concurrent runs of a particular job / task. This can be overridden at task level. Default value is set to 25
  • poke_interval: ?
  • cronProcessor:
    • start: If set to true, cron processes will start along with server
    • run_interval: Provide interval time between two cron processes in milliseconds
 log: {
        create_log_files: true,
        timestamp_format: 'dd-MMM hh:mm:ss',
        log_folder_format: 'yy-MMM-dd'
    }
  • create_log_files: If set to true, log files will be generated in the Runtime/Log folder
  • timestamp_format: Provide timestamp to be printed in log files
  • log_folder_format: Provide log folder name format
rpa: {
        web: {
            browser_type: 'chromium',
            headless: true,
            accept_downloads: false,
            store_state: false,
        }
    }
  • browser_type: Set browser to be used for Web RPA. Options available are chromium, firefox and webkit
  • headless: If set to true, web RPA will run in headless mode. User will not be able to see the browser when the RPA operation is being performed
  • accept_downloads: If set to true, files can be downloaded during RPA process
  • store_state: If set to true, the RPA context information will be stored into file system permanently. This will have to be managed manually. If set to false, it will be stored to temp folder and will be deleted as the RPA run gets over

mailgen: {
        theme: 'default',
        product: {
            name: 'Actionabl',
            link: 'https://actionabl.ai',
            logo: 'https://actionabl.ai/public_asset/assets/logo.svg'
        }
    }

Mailgen is a Node js package that generates clean, responsive HTML e-mails for sending transactional mail.

  • theme: Provide template to be used for generating HTML e-mails
  • product: Details about product to be configured here
    • name: Name of the product which will be used in email template
    • link: Link to product website
    • logo: Link for product logo
pyUtils: {
        endpoint: 'http://localhost:5000/'
    }
  • endpoint: This is applicable when the process uses ML tasks and this parameter will define the end point for the ML server
